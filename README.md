# react-xlsx — chart round-trip fidelity repro

Minimal reproduction showing that re-serializing a chart-containing workbook
through [`@extend-ai/react-xlsx`](https://github.com/extend-hq/react-xlsx) does
not preserve the chart. Both export paths are affected:

- `controller.workbook.saveXlsxBytes()` — the raw WASM-engine serialize
- `controller.exportXlsx()` — the library's own export (full image/drawing merge)

This blocks any "load → edit → save" feature, because **even a zero-edit
round-trip degrades the chart**.

## Run

```bash
npm install
npm run dev        # http://localhost:5173
```

Then:

1. **Load a real Excel-generated `.xlsx` that contains a chart** via the file
   picker (see the caveat below — the bundled sample does not work).
2. Confirm the page shows **charts parsed by engine: N (N > 0)** and the chart
   renders in the viewer.
3. Click **Download saveXlsxBytes (raw)** and **Download exportXlsx (merged)**.
4. Open both downloads next to the original in Excel / Numbers / Google Sheets.

**Expected:** the chart and its colors match the original.
**Actual:** chart colors are lost; depending on the file the chart part itself
is dropped.

## Verified observation (react-xlsx@0.12.1)

Re-serializing through the page and unzipping the output (the original sample's
chart carries three explicit `srgbClr` series colors in `xl/charts/chart1.xml`):

| Path | `xl/charts/*` in output | series `srgbClr` colors |
| --- | --- | --- |
| Original | `chart1.xml` | `FF0000`, `00B050`, `0070C0` |
| `saveXlsxBytes()` (raw) | **(none)** | **(none)** |
| `exportXlsx()` (merged) | **(none)** — `xl/drawings/drawing1.xml` is re-injected, but the `xl/charts/chart1.xml` it references is not | **(none)** |

The merge re-attaches the drawing layer (`xl/drawings/`) but not the chart parts
(`xl/charts/`) or theme, so the chart relationship dangles and the chart is lost.

First observed in our app on **0.10.2**, where the chart *did* survive but
rendered with the **wrong colors** (legend/series colors fell back to defaults)
— a less severe variant of the same fidelity loss. Bump the version in
`package.json` to compare releases.

## ⚠️ The bundled sample does not exercise the bug

`public/sample-chart.xlsx` is generated by `scripts/generate-sample.py`
(openpyxl). The engine does **not** parse its chart
(`controller.charts.length === 0`), so it neither renders nor round-trips — it
is only a placeholder/scaffold. **Use a chart file authored by Excel** (which
the engine does parse) via the file picker to reproduce the bug. Drop such a
file in as `public/sample-chart.xlsx` to make it the default.

## How the engine handles the wasm

`@extend-ai/react-xlsx@0.12.1` ships the wasm at
`dist/duke_sheets_wasm_bg.wasm` but does not export it as a package subpath, so
`scripts/copy-wasm.mjs` copies it into `public/` (run automatically by
`predev` / `prebuild`) and `src/main.tsx` points the engine at it with
`setWasmSource('/duke_sheets_wasm_bg.wasm')`. Vite also needs
`worker: { format: 'es' }` (the engine spawns a code-splitting Web Worker).

## Layout

```
src/main.tsx            setWasmSource + mount
src/App.tsx             file picker, viewer, both download buttons
scripts/copy-wasm.mjs   stage wasm into public/
scripts/generate-sample.py   regenerate the (placeholder) openpyxl sample
public/sample-chart.xlsx     placeholder chart workbook
```
